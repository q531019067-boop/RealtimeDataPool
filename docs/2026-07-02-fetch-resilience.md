# 2026-07-02 抓取韧性与反爬优化记录

## 背景

代码审查发现多源降级、启动调度、盘外抓取、API 限流关闭和数据保留配置存在偏差；同时请求层只重试 5xx，未对明确的 HTTP 429 限流信号做专门处理。

## 本次修改

- 多源健康度改为 `有效报价数 / 目标股票池总数`。主源覆盖率低于 70% 时进入下一数据源；全部数据源均不达标时返回有效数据最多的一组。
- HTTP 429 立即启用 penalty jitter，并将当前并发减半；重试优先遵守 `Retry-After`，最大等待 30 秒。
- p95 恢复过早但并发恢复仍在冷却时，会在正常状态累计满 200 个请求后恢复初始并发，避免永久停留在降级并发。
- 429、5xx 和网络异常使用带随机量的指数退避，避免多个请求同步重试；最终失败前释放响应连接。
- scheduler warm-up 后等待一个 basic 周期，消除启动时连续两次全市场抓取。
- orderbook 在失败、无目标等情况下也记录本次尝试时间，避免每个 30 秒 basic tick 重试腾讯。
- `fetch_out_of_session=false` 时，basic 和 orderbook 都在盘外停止；默认配置同步改为关闭盘外抓取。
- snapshot 清理使用 `storage.retention_days`，不再固定为 7 天。
- `RDP_RATE_LIMIT_PER_MIN=0` 正确表示关闭 API 限流，不再触发空队列异常。

## 验证范围

- 新增多源低覆盖降级、全部源降级时选择最佳结果、零值 API 限流和 retention 配置测试。
- 只允许运行单元测试、静态检查和不超过 30 秒的 smoke；禁止用 live 数据源验证。

## 运维观察点

- 关注日志中的 `DEGRADED`、`HTTP 429`、`THROTTLE detected` 和并发变化。
- 若 429 仍持续出现，优先降低 `concurrency_per_source` 或增大 `jitter_ms`，不要增加重试次数。
- `fetch_out_of_session=true` 会显著增加全天请求量，只应在明确需要盘外快照时开启。
